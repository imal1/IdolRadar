# IdolRadar 功能设计文档（MVP）

日期：2026-07-22
状态：已评审通过
后续产物：PRD、UI 设计文档、实现计划（本文档为其输入）

## 1. 背景与目标

IdolRadar 是一个面向追星族的微信小程序。用户像拥有一台"雷达"，实时追踪自己喜欢的某一位 idol 的全网动态。

MVP 目标：以最小功能集合跑通完整闭环——用户关注一位 idol，系统持续抓取其动态，有新动态时通过微信订阅消息推送给用户，用户点推送回到小程序查看。

## 2. 范围

### 2.1 MVP 包含

- 微信静默登录（仅使用 openid，无注册流程）
- 从管理员预置的 idol 库中选择并关注 1 位 idol
- 动态来源：RSS 第三方订阅源（管理员通过受保护的 seed/import CLI 配置）
- 定时抓取 RSS、解析、去重、入库
- 雷达首页动态流（时间倒序、下拉刷新、分页）
- 微信订阅消息推送新动态
- 我的页：关注管理、推送授权管理

### 2.2 MVP 不包含（明确排除）

- 动态详情页（仅聚焦雷达流，条目展示摘要与原文链接）
- 微博、抖音等平台 API 直接接入（后续版本）
- 多 idol 关注（单用户同时只能关注 1 位）
- 管理端界面（MVP 使用受保护的 seed/import CLI）
- 收费功能（仅保留数据字段占位）
- 社区评分、时区推测、额度统计等参考产品中的复杂功能

## 3. 技术方案

选定方案：自建 Java 21 + Spring Boot 4 API、PostgreSQL、Redis、独立定时 Worker、微信订阅消息 HTTP OpenAPI。

选择理由：身份、数据、任务、日志、备份与部署完全可控；不绑定 CloudBase；Java 虚拟线程适合高并发 I/O，API 可无状态水平扩容。API 与 Worker 使用同一制品，MVP 保持模块化单体。

已排除：CloudBase（部署、可观测性与数据迁移受平台约束）；无后端直连第三方 RSS 转 JSON 服务（无法安全推送，不满足闭环）。

## 4. 用户闭环 SOP

1. 用户进入小程序 → `wx.login` 取得 code → 自建 API 调用微信 `code2Session` 获取 openid → 首访建档并签发自有会话 token
2. 无关注时进入选 idol 页 → 从预置库选择 1 位 idol 关注
3. 用户点击「去开启」后请求订阅消息授权（用户勾选"总是保持以上选择"后可减少重复操作）
4. 独立 worker 定时抓取 RSS → 发现新动态 → 向关注者发送订阅消息
5. 用户点击推送 → 直达雷达首页并定位最新动态 → 浏览动态流
6. 用户可在我的页更换 idol（旧关注自动解除）或重新授权推送

闭环判定：步骤 1–5 全部可走通即 MVP 达成。

## 5. 页面设计（共 3 页）

### 5.1 选 idol 页（首次引导 / 更换入口）

- 预置 idol 列表：头像、名字、简介、源数量
- 搜索框（按名字过滤，本地过滤即可）
- 点选即关注；已有关注时进入则为"更换"语义，确认后替换

### 5.2 雷达首页（核心页 / 推送落地页）

- 头部卡片:当前 idol 头像、名字、最近更新时间、雷达扫描状态视觉（装饰性，不做概率研判）
- 动态流：时间倒序列表，每条含标题、摘要、来源渠道标签、发布时间
- 条目操作：点击复制原文链接（MVP 不内嵌 web-view）
- 交互：下拉刷新、上滑分页（每页 20 条）
- 推送落地：点击订阅消息打开此页并滚动至最新动态

### 5.3 我的页

- 当前关注的 idol 与更换入口（跳选 idol 页）
- 推送订阅状态展示与重新授权按钮
- 关于/版本信息

## 6. 后端设计（自建服务）

### 6.1 服务

| 服务 | 触发方式 | 职责 |
|---|---|---|
| Spring Boot API | 小程序 HTTPS 调用 | 微信 code 登录、会话鉴权、关注/更换 idol、记录订阅额度、动态分页；Redis 跨实例限流 |
| `fetch-feeds` Worker | 宿主 cron/Kubernetes CronJob，每 30 分钟 | 遍历 RSS 源、SSRF 安全拉取、解析去重、写入 `posts`、调用进程内通知服务 |
| 通知服务 | Worker 进程内调用 | Redis 缓存并单飞刷新微信 access token；持久化 outbox；原子预扣额度；失败补偿；记录幂等投递 |

### 6.2 数据模型（PostgreSQL）

**idols**

| 字段 | 说明 |
|---|---|
| name | 名字 |
| avatar | 头像 URL |
| bio | 简介 |
| enabled | 启用状态 |

**sources**

| 字段 | 说明 |
|---|---|
| idolId | 所属 idol |
| rssUrl | RSS 地址 |
| channel | 渠道名（用于来源标签，如微博/ins/B站） |
| enabled | 启用状态 |
| lastFetchAt / lastFetchStatus | 最近抓取时间与结果 |

**posts**

| 字段 | 说明 |
|---|---|
| idolId | 所属 idol |
| sourceId | 来源 source |
| title | 标题 |
| summary | 摘要 |
| link | 原文链接（唯一索引，去重键） |
| publishedAt | 发布时间 |
| fetchedAt | 抓取时间 |

**users**

| 字段 | 说明 |
|---|---|
| openid | 微信 openid（唯一约束；仅由服务端 `code2Session` 获取） |
| idolId | 当前关注的 idol（可空） |
| subscribeQuota | 订阅消息剩余额度计数 |
| subscribedAt | 最近授权时间 |
| phone | 手机号（占位，MVP 不录入，为收费功能预留） |
| billing | 付费/额度记录（占位，MVP 不录入） |

### 6.3 推送机制

- 使用微信小程序订阅消息（一次授权发一条）
- 用户点击「去开启」后调用 `wx.requestSubscribeMessage` 请求授权，成功则 `subscribeQuota +1`；用户勾选"总是保持以上选择"后可减少重复操作
- `sendNotify` 仅向 `subscribeQuota > 0` 的关注者发送，发送成功后 `subscribeQuota -1`
- 同一轮抓取发现多条新动态时，每用户只推送 1 条（最新一条），避免额度快速耗尽

### 6.4 错误处理

- 单个 RSS 源抓取/解析失败不影响其他源，写入 `lastFetchStatus` 供控制台排查
- 订阅消息使用持久化状态机：确定未发出则退避重试，用户拒收则终止，结果不确定则不自动重试
- 新动态与 `notification_outbox` 同事务；Worker 中断后按 lease 恢复，delivery 幂等键防止续跑重发
- RSS 条目缺字段时的降级：无摘要用标题、无发布时间用抓取时间

## 7. 管理端

MVP 不做管理界面。idol 与 RSS 源通过受保护的 seed/import CLI 维护；抓取失败状态通过数据库与 worker 日志查看。

## 8. 测试要点

- API/worker 单测：登录鉴权、RSS 解析（正常/缺字段/坏 XML）、去重、额度原子扣减与失败补偿
- 闭环手测脚本：新用户首访 → 选 idol → 授权 → 模拟新动态入库 → 收到推送 → 落地雷达首页
- 边界：更换 idol 后旧动态不再展示、推送不再发送；额度为 0 时不发送

## 9. 未来扩展（不在本期）

- 微博、抖音等平台 API 接入（worker 内新增 fetcher，数据模型已兼容多渠道）
- 动态详情页与站内浏览
- 多 idol 关注
- 管理端界面与 idol 库自助申请
- 收费功能（users 表 phone/billing 字段已预留）
