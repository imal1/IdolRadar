## 1. 产品概述

### 1.1 一句话定义

IdolRadar 是一个面向追星族的微信小程序：用户"守护"一位 idol，系统持续抓取其全网动态，一有新动态就通过微信订阅消息提醒用户回来查看。

### 1.2 目标用户

- 核心用户：重度追星族（"蹲守小分队"），希望第一时间知道自己 idol 的动态
- 使用场景：碎片时间刷动态；收到"她营业了"推送后立即点进来看

### 1.3 MVP 目标（闭环判定）

以最小功能集合跑通完整闭环，以下 5 步全部走通即达成：

1. 用户进入小程序，静默登录成功
2. 从预置库选择并守护 1 位 idol
3. 完成订阅消息授权
4. 云端抓到新动态后，用户收到微信订阅消息
5. 用户点推送直达雷达首页，看到最新动态

### 1.4 成功指标（MVP 阶段观测项）

| 指标 | 说明 |
|---|---|
| 闭环转化率 | 新用户中完成"选 idol + 授权"的比例 |
| 推送到达后回访率 | 收到推送后 24h 内打开小程序的比例 |
| 抓取健康度 | RSS 源抓取成功率（`lastFetchStatus`） |

## 2. 范围

### 2.1 包含

- 微信静默登录（仅 openid，无注册流程）
- 从管理员预置 idol 库中选择并守护 1 位 idol（单选，换人替换）
- RSS 源定时抓取（每 30 分钟）、解析、按原文链接去重、入库
- 雷达首页动态流：时间倒序、下拉刷新、上滑分页（每页 20 条）
- 微信订阅消息推送新动态（每轮每用户最多 1 条）
- 我的页：守护管理、推送授权管理、关于信息

### 2.2 不包含（明确排除）

- 动态详情页（条目仅展示摘要 + 复制原文链接，不内嵌 web-view）
- 微博、抖音等平台 API 直接接入
- 多 idol 同时守护
- 管理端界面（云开发控制台手工维护）
- 收费功能（仅保留数据字段占位）
- 社区评分、时区推测、额度统计等复杂功能

## 4. 页面需求（共 3 页）

页面结构与文案以 Figma 设计稿为准（node-id 见各节），390×844 基准。

### 4.1 雷达首页（Figma `1:2` "雷达页 · 动态追踪"）

核心页 / 推送落地页。导航标题 **IdolRadar**。

#### 4.1.1 idol 头部卡片

- 内容：头像（含在线状态圆点）、名字、"营业中"徽章、感应状态行（如"感应中 · 最近更新 18 分钟前"）、右侧"换她"按钮（跳选 idol 页）
- 统计行 3 格：今日动态数 / 信号源数 / 信号强度（"满格"，装饰性文案，不做概率研判）
- 数据来源：`user` 云函数返回当前守护 idol 信息 + 动态统计

#### 4.1.2 蹲守提醒 banner

- 文案：「开启蹲守提醒 / 她一营业，微信马上叫你」+「去开启」按钮
- 仅当用户订阅额度为 0 或未授权时展示；点击触发 `wx.requestSubscribeMessage`
- 授权成功后 banner 隐藏

#### 4.1.3 "刚蹲到的"置顶卡片

- 区块标题「刚蹲到的」，右侧说明文案「每 30 分钟感应一次」
- 展示最新一条动态：「蹲到了！」标签 + 渠道标签（微博/ins/B站…）+ 相对时间 + 标题 + 摘要 + 「复制原文链接」按钮
- 推送落地时滚动定位到此卡片

#### 4.1.4 "之前的动态"列表

- 区块标题「之前的动态」
- 每条：渠道标签、相对时间（今天内"N 分钟/小时前"，更早"昨天 20:14"/"前天 22:40"格式）、标题、摘要
- 交互：下拉刷新拉最新；上滑加载下一页，每页 20 条；列表底部文案「上滑加载更多 · 每页 20 条」
- 点击条目：复制原文链接（toast 提示复制成功），不跳详情

#### 4.1.5 空状态与首访

- 无守护对象：自动跳转选 idol 页
- 有守护但暂无动态：展示空态提示（等待第一次抓取）

### 4.2 选 idol 页（Figma `1:3` "选 idol 页"）

首次引导 / 更换入口。导航标题 **选择守护对象**。

- 页头文案：「今天想蹲谁？」+ 副标题「同时只能守护一位，换人后旧动态会清空」
- 搜索框（placeholder「输入名字找找看」）：按名字本地过滤
- idol 卡片列表：头像、名字、简介、信号源数量、右侧操作按钮
  - 当前守护对象：名字旁「在守护」徽章，按钮态「守护中」（禁用）
  - 其他：按钮「换成她」；无守护时按钮语义为"守护"
- 更换需确认弹窗（提示旧动态清空、推送切换），确认后替换并返回雷达首页
- 页脚文案：「想蹲的人不在里面？之后会开放自助申请~」

### 4.3 我的页（Figma `1:4` "我的页"）

导航标题 **我的**。

- 用户卡片：默认头像 + 「微信用户」+「静默登录 · 无需注册」
- 「正在守护」卡片：idol 头像、名字、「已守护 N 天 · N 个信号源」、「换一位」按钮（跳选 idol 页）
- 「蹲守提醒」卡片：授权状态（「提醒未开启」/已开启 + 剩余额度语义）、「去开启」按钮、小提示文案「授权时勾选『总是保持以上选择』，以后就不用每次点啦」
- 关于列表：「关于 IdolRadar」（版本号 v0.1.0）、「动态来源说明」（RSS 订阅源）
- 页脚 slogan：「追星是双向奔赴的浪漫 ♡」

## 5. 功能需求明细

### 5.1 登录（FR-1，P0）

- 进入小程序 `wx.login` 静默登录；云函数上下文取 openid，首访自动建档
- 无任何注册/手机号录入流程
- 验收：新设备首次打开即可正常使用全部功能

### 5.2 守护关系（FR-2，P0）

- 单用户同时仅 1 个守护对象；选择即守护，再选即替换
- 更换后：雷达首页只展示新 idol 动态，旧 idol 推送立即停止
- 验收：换人后旧动态不再展示、旧推送不再发送

### 5.3 动态抓取（FR-3，P0）

- 定时触发器每 30 分钟执行 `fetchFeeds`：遍历启用 RSS 源 → 拉取解析 → 按原文链接（唯一索引）去重 → 新条目写入 `posts` → 触发推送
- 降级：无摘要用标题；无发布时间用抓取时间
- 单源失败不影响其他源，结果写 `lastFetchStatus`
- 验收：坏 XML / 缺字段源不阻塞整轮抓取；同链接不重复入库

### 5.4 订阅消息推送（FR-4，P0）

- 用户点击「去开启」后调用 `wx.requestSubscribeMessage`，成功 `subscribeQuota +1`；引导勾选「总是保持以上选择」
- `sendNotify` 仅向守护该 idol 且 `subscribeQuota > 0` 的用户发送；成功后额度 -1
- 消息内容：idol 名、动态标题、时间；点击直达雷达首页并定位最新动态
- 同轮多条新动态：每用户只推最新 1 条
- 失败（额度不足/拒收）记录并跳过，不重试阻塞
- 验收：额度为 0 不发送；一轮 3 条新动态用户只收 1 条推送

### 5.5 动态流（FR-5，P0）

- 时间倒序（publishedAt），下拉刷新、上滑分页每页 20 条
- 条目字段：渠道标签、标题、摘要、相对时间
- 点击复制原文链接
- 验收：分页无重复/遗漏；刷新拉到最新

## 6. 技术方案（摘要）

微信云开发一体化：云函数（`fetchFeeds` / `sendNotify` / `user`）+ 云数据库（`idols` / `sources` / `posts` / `users` 4 集合）+ 定时触发器 + 订阅消息。字段定义见[功能设计文档 §6](MVP-功能设计.md)。

选择理由：免备案、免运维、原生定时与推送能力、单人可维护；后续可在云函数内扩展微博/抖音 fetcher，用户量增大可迁自建后端。

## 7. 非功能需求

| 项 | 要求 |
|---|---|
| 抓取时效 | 新动态从发布到推送 ≤ 35 分钟（30 分钟周期 + 处理时间） |
| 数据一致性 | `posts.link` 唯一索引兜底去重 |
| 可观测性 | `sources.lastFetchAt/lastFetchStatus` 可在控制台排查 |
| 合规 | 不存储手机号等敏感信息（字段仅占位不录入）；动态内容仅存标题/摘要/链接 |

## 8. 管理端

MVP 不做界面。idol 与 RSS 源由管理员在云开发控制台手工录入维护；抓取失败通过控制台查看。

## 9. 测试要点

- 单测：RSS 解析（正常/缺字段/坏 XML）、去重、额度扣减
- 闭环手测：新用户首访 → 选 idol → 授权 → 模拟新动态入库 → 收到推送 → 落地雷达首页定位最新动态
- 边界：换 idol 后旧动态/旧推送清理；额度 0 不发送；空动态流展示
