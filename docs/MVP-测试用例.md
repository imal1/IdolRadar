# IdolRadar 测试用例文档（MVP）

日期：2026-07-22
状态：草稿（待评审）
上游输入：[产品需求文档](MVP-产品需求PRD.md)、[功能设计文档](MVP-功能设计.md)
测试范围：MVP 全部功能（登录、选 idol、动态抓取、订阅推送、动态流、我的页）

优先级说明：P0 = 闭环必测（任一失败即 MVP 不达成）；P1 = 主功能；P2 = 边界与体验。

---

## 1. 登录与建档

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| LOGIN-01 | P0 | 新设备/新微信号首次打开 | 打开小程序 | 无任何注册/登录界面；静默登录成功；云数据库 `users` 新增一条记录（openid 唯一，idolId 为空） |
| LOGIN-02 | P0 | 老用户（已建档） | 再次打开小程序 | 不重复建档；读取已有守护关系直接进入对应页面 |
| LOGIN-03 | P1 | 新用户（无守护对象） | 打开小程序 | 自动跳转选 idol 页 |
| LOGIN-04 | P2 | 网络异常 | 断网打开小程序 | 有失败提示，不白屏；恢复网络后重试可用 |

## 2. 选 idol 页

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| IDOL-01 | P0 | 新用户，idol 库已预置数据 | 进入选 idol 页 | 列表展示全部启用 idol：头像、名字、简介、信号源数量；按钮为守护语义 |
| IDOL-02 | P0 | 同上 | 点选一位 idol | 守护成功；`users.idolId` 更新；跳转雷达首页并展示该 idol |
| IDOL-03 | P1 | 已守护苏念 | 再次进入选 idol 页 | 苏念卡片显示「在守护」徽章，按钮「守护中」置灰不可点；其他显示「换成她」 |
| IDOL-04 | P1 | 已守护苏念 | 点其他 idol 的「换成她」 | 弹确认弹窗（提示旧动态清空、推送切换）；确认后替换成功并返回雷达首页 |
| IDOL-05 | P1 | 同上 | 确认弹窗中点取消 | 守护关系不变，停留在选 idol 页 |
| IDOL-06 | P1 | idol 库 ≥ 2 条 | 搜索框输入名字关键字 | 本地实时过滤；清空关键字恢复全列表 |
| IDOL-07 | P2 | — | 搜索无匹配结果 | 展示空结果提示，不报错 |
| IDOL-08 | P2 | 控制台将某 idol `enabled` 置 false | 刷新选 idol 页 | 该 idol 不在列表中出现 |

## 3. 更换 idol 的数据边界

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| SWITCH-01 | P0 | 守护苏念且已有其动态流 | 换成林晚，回雷达首页 | 只展示林晚动态；苏念旧动态不再展示 |
| SWITCH-02 | P0 | 同上，用户有订阅额度 | 换人后模拟苏念新动态入库 | 用户不再收到苏念的推送 |
| SWITCH-03 | P1 | 换人后林晚有新动态 | 触发抓取与推送 | 用户正常收到林晚的推送 |
| SWITCH-04 | P2 | 换人后 | 查看我的页「正在守护」卡片 | 展示林晚，守护天数从换人日重新计算 |

## 4. RSS 抓取与解析（云函数 `fetchFeeds`，单测 + 集成）

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| FETCH-01 | P0 | 配置 1 个正常 RSS 源 | 手动触发/等定时器执行 | 新条目写入 `posts`（idolId、sourceId、title、summary、link、publishedAt、fetchedAt 齐全）；`sources.lastFetchAt/lastFetchStatus` 更新为成功 |
| FETCH-02 | P0 | `posts` 已含链接 X 的条目 | 再次抓取含链接 X 的源 | 链接 X 不重复入库（唯一索引兜底） |
| FETCH-03 | P0 | 源返回坏 XML | 触发抓取 | 该源 `lastFetchStatus` 记录失败；同轮其他源正常抓取入库 |
| FETCH-04 | P1 | 条目缺摘要 | 触发抓取 | `summary` 降级取标题，正常入库 |
| FETCH-05 | P1 | 条目缺发布时间 | 触发抓取 | `publishedAt` 降级取抓取时间，正常入库 |
| FETCH-06 | P1 | 源请求超时/404 | 触发抓取 | 记录失败状态，不阻塞其他源，不写脏数据 |
| FETCH-07 | P2 | 源 `enabled` 置 false | 触发抓取 | 该源被跳过，不请求 |
| FETCH-08 | P2 | 定时触发器已配置 | 等待 30 分钟周期 | 定时器按期自动执行，控制台可见执行日志 |

## 5. 订阅消息授权与额度

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| SUB-01 | P0 | 新用户未授权 | 进入小程序，触发授权请求，点允许 | `subscribeQuota +1`；`subscribedAt` 更新；雷达首页「开启蹲守提醒」banner 隐藏 |
| SUB-02 | P1 | 授权弹窗出现 | 点拒绝 | 额度不变；banner 继续展示；不重复死缠弹窗（遵循微信限制） |
| SUB-03 | P1 | 用户勾选「总是保持以上选择」+允许 | 之后每次进入小程序 | 静默累积额度，无弹窗打扰 |
| SUB-04 | P1 | 额度为 0 | 查看雷达首页与我的页 | banner 展示「去开启」；我的页「蹲守提醒」显示未开启态 |
| SUB-05 | P2 | 已授权 | 我的页点「去开启」重新授权 | 可再次发起授权，额度累加 |

## 6. 推送发送（云函数 `sendNotify`）

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| PUSH-01 | P0 | 用户守护苏念，`subscribeQuota ≥ 1` | 模拟苏念新动态入库触发推送 | 收到微信订阅消息（含 idol 名、动态标题、时间）；该用户 `subscribeQuota -1` |
| PUSH-02 | P0 | 用户 `subscribeQuota = 0` | 同上触发推送 | 不发送；不报错；额度保持 0 |
| PUSH-03 | P0 | 同轮抓取产生 3 条新动态 | 触发推送 | 每用户只收到 1 条（最新一条）；额度只扣 1 |
| PUSH-04 | P0 | 用户收到推送 | 点击订阅消息 | 直达雷达首页；滚动定位到「刚蹲到的」最新动态 |
| PUSH-05 | P1 | 用户 A 守护苏念、用户 B 守护林晚 | 苏念新动态入库 | 只有 A 收到推送，B 不收 |
| PUSH-06 | P1 | 发送 API 返回失败（拒收/额度） | 触发推送 | 记录失败并跳过该用户；不重试阻塞；其他用户正常发送 |
| PUSH-07 | P2 | 无新动态的抓取轮 | 定时器执行 | 不发送任何推送 |

## 7. 雷达首页动态流

| 用例 ID | 优先级 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| FEED-01 | P0 | 有 ≥ 1 条动态 | 进入雷达首页 | 头部卡片展示 idol 信息与统计；「刚蹲到的」展示最新一条；「之前的动态」时间倒序排列 |
| FEED-02 | P0 | 有 ≥ 21 条动态 | 上滑到底 | 加载第 2 页（每页 20 条）；无重复、无遗漏；底部显示「上滑加载更多 · 每页 20 条」 |
| FEED-03 | P1 | 页面停留中，后台入库新动态 | 下拉刷新 | 列表头部出现新动态 |
| FEED-04 | P1 | 任一动态条目 | 点「复制原文链接」/点条目 | 剪贴板为原文链接；toast 提示复制成功；不发生页面跳转 |
| FEED-05 | P1 | 各时间段动态 | 检查时间展示 | 当天显示「N 分钟前/N 小时前」；更早显示「昨天 20:14」「前天 22:40」式格式 |
| FEED-06 | P1 | 条目含渠道 | 检查来源标签 | 标签与 `sources.channel` 一致（微博/ins/B站/行程…） |
| FEED-07 | P2 | 守护后暂无任何动态 | 进入雷达首页 | 展示空态提示，不报错、不白屏 |
| FEED-08 | P2 | 动态刚好 20 条 | 上滑到底 | 第 2 页为空时正确终止，无死循环加载 |

## 9. 端到端闭环（验收主线）

| 用例 ID | 优先级 | 步骤 | 预期结果 |
|---|---|---|---|
| E2E-01 | P0 | 新用户首次打开 → 自动进选 idol 页 → 选苏念 → 允许订阅授权 → 管理员模拟新动态入库（或等定时抓取）→ 微信收到推送 → 点推送进入雷达首页 | 全链路无阻断；落地页定位到最新动态；`users`/`posts` 数据正确 | 

E2E-01 为 MVP 达成判定用例：失败即视为闭环未打通。

## 10. 测试环境与数据准备

- 云开发测试环境独立于生产，预置 ≥ 2 位 idol（各配 ≥ 1 个 RSS 源，其中一个可控源用于模拟新动态）
- 可控 RSS 源建议：自建静态 XML（可手动追加条目模拟"新动态"），另备一个坏 XML 地址、一个 404 地址
- 订阅消息需真机测试（开发者工具不支持真实推送）；至少 2 个测试微信号（覆盖 PUSH-05）
- 云函数单测覆盖：RSS 解析（正常/缺字段/坏 XML）、去重、额度扣减（对应 FETCH-01~05、PUSH-01~03）
